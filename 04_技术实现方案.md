# 《王者荣耀·长城试炼：百里守约 AI 战术训练系统》技术实现方案

## 一、技术栈建议
### 1. 核心技术栈
- 游戏引擎：Phaser 3
- 开发语言：TypeScript 优先，JavaScript 可作为备选
- 构建工具：Vite
- UI 辅助：原生 DOM + 简洁 CSS，或 Vue/React 仅用于外围页面
- 数据管理：本地状态管理 + JSON 配置
- 接口层：Node.js 轻量代理服务或云函数
- 部署方式：静态前端部署 + AI 接口服务部署

### 2. 建议组合
最推荐组合为：Phaser 3 + TypeScript + Vite + Node.js API 代理

该组合优点在于：
- 上手快，适合比赛短周期开发
- 便于快速搭建 2D 游戏场景
- 代码结构清晰，适合多人协作
- 浏览器运行方便现场演示与录屏

## 二、为什么推荐 Phaser 3
### 1. 适合 2D / 2.5D demo
本项目的重点是玩法闭环和 AI 融合，不是追求重型 3D 表现。Phaser 3 足以支撑角色移动、敌人巡逻、拾取、战斗、UI 覆盖层和场景切换等需求。

### 2. 开发速度快
Phaser 3 对精灵、碰撞、动画、输入、相机和场景管理支持成熟，便于在 3 至 4 周内搭建可演示版本。

### 3. 浏览器适配友好
比赛演示通常依赖浏览器运行，Phaser 3 输出稳定、部署轻量，适合在本地或云端快速展示。

### 4. 模块化程度适中
对于 3 人团队而言，Phaser 3 既不会过重，也不会过于原始，适合作为兼顾效率和扩展性的方案。

## 四、核心模块划分
### 1. 场景模块
- `BootScene`：资源预加载
- `StartScene`：启动页
- `TrainingRoomScene`：训练室入口
- `BriefingScene`：任务简报页
- `MissionScene`：主游戏场景
- `ResultScene`：结算页
- `ReviewScene`：AI 复盘页

### 2. 实体模块
- `Player`：百里守约角色行为
- `Enemy`：魔种基础行为
- `ItemDrop`：可拾取资源

### 3. 系统模块
- `CombatSystem`：攻击判定、伤害结算、技能冷却
- `InventorySystem`：背包容量、拾取、替换、使用逻辑
- `MissionSystem`：任务目标与进度统计
- `RiskSystem`：风险等级评估与事件触发
- `ExtractionSystem`：撤离点开放与完成判定
- `AISystem`：状态组装、AI 调用节流、结果派发

### 4. UI 模块
- 顶部 HUD
- 背包面板
- AI 教练提示区
- AI 问答弹层
- 结算与复盘展示页

## 五、场景管理结构
### 1. 场景切换顺序
`BootScene -> StartScene -> TrainingRoomScene -> BriefingScene -> MissionScene -> ResultScene -> ReviewScene`

### 2. 数据流转建议
- 全局 `runContext` 保存当前任务状态
- 场景切换时传递最小必要数据
- 整局结束后生成 `reviewPayload`

### 3. 关键切换点
- 训练室点击“开始训练”进入简报页
- 简报页点击“进入模拟战场”进入主场景
- 主场景完成撤离后进入结算页
- 结算页点击“查看 AI 复盘”进入复盘页

## 六、游戏主循环
在 `MissionScene` 中建议采用如下更新逻辑：

1. 接收输入
- 处理角色移动、攻击、冲刺、交互和打开面板操作

2. 更新角色状态
- 位置更新
- 动画更新
- 冷却时间更新

3. 更新敌人状态
- 巡逻
- 追击
- 攻击判定

4. 处理碰撞与交互
- 玩家与资源碰撞
- 玩家与敌人碰撞
- 玩家与撤离点碰撞

5. 更新系统状态
- 风险等级
- 任务进度
- 背包占用
- AI 触发条件

6. 渲染 UI
- HUD 数值
- AI 提示
- 任务目标

## 七、角色控制模块
### 1. 输入方式
- `WASD` 或方向键移动
- 鼠标或按键触发瞄准与攻击
- `Space` 冲刺
- `E` 拾取 / 交互
- `I` 打开背包
- `Q` 打开 AI 问答

### 2. 角色状态机建议
- Idle
- Move
- Aim
- Attack
- Dash
- Hurt
- Interact

### 3. 实现重点
- 移动必须流畅，保证 demo 观感
- 攻击反馈要清晰，最少包含音效或动画变化
- 交互半径要稳定，减少拾取失败体验

## 八、战斗模块
### 1. 战斗方式
- 基础普攻
- 蓄力瞄准攻击

### 2. 战斗流程
- 锁定方向
- 发射投射物
- 命中判定
- 扣血与击退
- 敌人死亡或继续追击

### 3. MVP 要求
- 至少 2 类敌人
- 至少 2 种攻击反馈
- 明确生命值变化

## 九、敌人模块
### 1. 敌人行为层级
- 默认巡逻
- 发现玩家后追击
- 接近后攻击
- 失去目标后返回巡逻

### 2. 数据驱动建议
通过 `enemies.json` 配置以下字段：
- 名称
- 生命值
- 移动速度
- 警戒半径
- 攻击伤害
- 掉落概率

### 3. 可扩展性
后续可继续添加精英魔种和远程压制型敌人，但 MVP 阶段不建议同时做太多种类。

## 十、背包模块
### 1. 数据结构建议
```json
{
  "capacity": 8,
  "slots": [
    { "itemId": "food_pack", "count": 1 },
    { "itemId": "medkit", "count": 1 },
    { "itemId": "ammo", "count": 2 }
  ]
}
```

### 2. 功能要求
- 拾取资源
- 背包满提示
- 丢弃替换
- 查看物品说明
- 使用部分消耗品

### 3. UI 要点
- 格子清晰
- 关键任务资源高亮
- 展示容量占用

## 十一、撤离模块
### 1. 状态条件
- 达到最低任务条件
- 进入撤离区域
- 保持若干秒不被打断

### 2. 技术实现
- 撤离点使用碰撞区域
- 倒计时使用定时器
- 中断条件由敌人攻击或离开区域触发

### 3. 演示建议
撤离时增加屏幕提示、进度条与警报音效，增强紧张感。

## 十二、AI 调用模块
### 1. 模块职责
- 收集游戏状态
- 格式化请求体
- 控制调用频率
- 接收结果并映射到 UI

### 2. 调用时机
- 简报页加载完成时请求局前 AI 简报
- 主游戏场景每隔 20 至 30 秒或重大事件触发请求局内教练
- AI 问答弹层提交问题时请求问答接口
- 结算后请求 AI 复盘接口

### 3. 接口设计建议
`POST /api/ai/briefing`
`POST /api/ai/coach`
`POST /api/ai/qa`
`POST /api/ai/review`

### 4. 返回格式建议
统一使用结构化 JSON，前端再渲染为文本组件，避免直接拼接复杂字符串。

## 十三、数据存储结构
### 1. 本地静态配置
- 任务配置
- 敌人配置
- 物品配置
- 地图资源点配置

### 2. 运行时状态
- 当前局角色状态
- 背包内容
- 已拾取资源
- 已击败敌人
- 风险等级
- 撤离状态

### 3. 持久化建议
MVP 阶段可使用 `localStorage` 保存：
- 上次训练评分
- 玩家偏好设置
- AI 复盘记录摘要

## 十四、API 接入设计
### 1. 推荐架构
浏览器前端不直接暴露密钥，建议通过一个轻量中转层处理 AI 请求。

### 2. 服务端职责
- 接收前端状态数据
- 补充系统级 prompt
- 转发到腾讯元器或相关 AI 服务
- 返回格式化结果

### 3. 容错策略
- 调用失败时使用预置规则文案兜底
- 超时时显示“训练室信号不稳定，请稍后重试”
- 局内建议接口必须有节流与缓存，避免卡顿

## 十五、demo 部署建议
### 1. 本地演示
- 使用 Vite 本地启动
- 确保素材路径和接口代理已配置
- 准备离线兜底版，避免现场网络波动影响

### 2. 线上部署
- 前端部署到静态托管平台
- API 代理部署到云服务器或云函数
- 将演示地址写入答辩材料与视频说明

### 3. 演示优化
- 首屏尽量控制在 5 秒内进入
- 主流程避免依赖复杂登录
- 预置一局演示存档或演示模式，方便现场稳定展示

## 十六、技术实现结论
本项目技术方案的重点是“轻量、稳定、可演示、可扩展”。Phaser 3 负责快速搭建游戏化核心体验，TypeScript 提供清晰的工程组织，AI 模块通过代理接口接入腾讯元器等能力。通过场景分离、模块拆分和数据结构化设计，3 人团队可以在较短周期内完成一个兼具可玩性与 AI 亮点的比赛 demo。
